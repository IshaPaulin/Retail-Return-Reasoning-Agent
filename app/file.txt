SECTION 1: WHAT THIS PROJECT IS
--------------------------------------------------------------------------------

This is a backend + frontend web application built for sellers on a retail
platform. The purpose of the system is to help sellers understand WHY their
products are being returned — using AI to analyse their own historical data and
generate actionable, evidence-grounded insights.

There is NO manual analysis. The AI agent does the reasoning automatically.
Sellers just log in, see their dashboard, and ask questions if they want more.

The system has four major pieces:
  1. Authentication (JWT-based login)
  2. Dashboard (AI-generated per-product insights shown as cards)
  3. Product Detail View (full breakdown for a single product)
  4. Chatbot (seller asks natural language questions, agent answers using data)

The AI backbone is Google Gemini (free tier), using native function calling.
The database is MongoDB. The backend is FastAPI (Python). The frontend is Streamlit.

SECTION 6: AUTHENTICATION (auth/)
--------------------------------------------------------------------------------

6.1 Hashing (hashing.py)
  - hash_password(plain: str) -> str         uses bcrypt
  - verify_password(plain: str, hashed: str) -> bool

6.2 JWT (jwt.py)
  Token payload contains:
    { "seller_id": "...", "exp": <unix timestamp> }

  Functions:
    - create_access_token(data: dict) -> str
      Signs the token with SECRET_KEY, sets expiry (e.g. 24 hours)
    - decode_access_token(token: str) -> dict
      Decodes and validates. Raises HTTPException 401 if invalid or expired.
    - get_current_seller(token: str = Depends(oauth2_scheme)) -> str
      FastAPI dependency. Extracts seller_id from validated token.
      Use this as a dependency on ALL protected routes.

6.3 Routes (routes.py)
  POST /auth/login
    - Request body: { "username": "...", "password": "..." }
    - Look up seller in sellers collection by username
    - Verify password with verify_password()
    - If valid: return { "access_token": "...", "token_type": "bearer" }
    - If invalid: raise HTTPException 401 with message "Invalid credentials"
    - This is the ONLY public endpoint. Everything else requires a valid JWT.


--------------------------------------------------------------------------------
SECTION 7: DATA ISOLATION — THE SCOPE GUARD
--------------------------------------------------------------------------------

CRITICAL RULE: Every database query MUST filter by seller_id.
The seller_id comes from the JWT token (via get_current_seller dependency).
It is NEVER passed in from the request body or query params by the user.

This means:
  - A seller can only ever see their own products, returns, orders, feedback.
  - Even if a seller guesses another seller's product_id, the query filter
    ensures no data from the other seller is returned.
  - The AI agent is NOT the security layer. The query filter is.

Every tool function signature includes seller_id as a parameter.
The seller_id is always injected from the validated JWT, not from user input.

Example pattern in every tool:
  db.returns.find({ "seller_id": seller_id, "product_id": product_id })

SECTION 8: API ENDPOINTS
--------------------------------------------------------------------------------

8.1 POST /auth/login      ← PUBLIC (no JWT required)
  Request:  { "username": str, "password": str }
  Response: { "access_token": str, "token_type": "bearer" }
  Errors:   401 if credentials are wrong

8.2 GET /dashboard        ← PROTECTED (JWT required)
  What it does:
    - Extracts seller_id from the JWT
    - Fetches all products belonging to that seller_id from the products collection
    - For each product, runs the agent pipeline (see Section 10)
    - Returns a list of product insight cards

  Response format (list):
  [
    {
      "product_id": "...",
      "product_name": "...",
      "return_signal": "High" | "Normal" | "Low",
      "primary_pattern": "...",
      "summary": "one-line AI-generated summary"
    },
    ...
  ]

8.3 GET /dashboard/product/{product_id}    ← PROTECTED (JWT required)
  What it does:
    - Extracts seller_id from JWT
    - Verifies the product_id belongs to the authenticated seller
      (if not found under that seller_id, return 403 or 404)
    - Runs the agent pipeline for that specific product
    - Returns the full structured insight object

  Response format:
  {
    "product_id": "...",
    "product_name": "...",
    "return_signal": "High" | "Normal" | "Low",
    "summary": "...",
    "root_cause": "...",
    "patterns": ["...", "..."],
    "evidence": { ... },
    "trend": { ... } or null,
    "suggestions": ["...", "...", "..."],
    "anomalies": ["..."] or []
  }

  Omit any key whose data is unavailable (do not return null fields,
  just leave the key out of the response entirely).

8.4 POST /chat            ← PROTECTED (JWT required)
  What it does:
    - Extracts seller_id from JWT
    - Receives the seller's natural language query
    - Runs the agent pipeline in chatbot mode (LLM decides which tools to call)
    - Returns a grounded, evidence-based answer

  Request:  { "query": "Why is Product X getting so many returns?" }
  Response: { "response": "...", "tools_used": ["tool_name", ...] }

  If the query is out of scope (not about returns, products, orders, or feedback):
  Response: {
    "response": "I can only assist with return, product, order, and feedback related questions.",
    "tools_used": []
  }

SECTION 9: TOOLS (app/tools/)
--------------------------------------------------------------------------------

Each tool is a standalone Python function that queries MongoDB directly.
Tools never call each other. Tools never call the LLM. Tools only fetch data.

All tools take seller_id as a parameter and filter by it on every query.
All tools return raw data (dicts/lists) — no summarization, no formatting.
If the queried data doesn't exist, tools return an empty list or empty dict.

TOOL LIST:

9.1 get_product_return_data(product_id: str, seller_id: str) -> list
  Query: db.returns.find({ seller_id, product_id })
  Returns: list of all return records for this product

9.2 get_return_reasons_breakdown(product_id: str, seller_id: str) -> dict
  Query: aggregate returns for this product grouped by return_reason field
  Returns: { "reason_1": count, "reason_2": count, ... }

9.3 get_customer_feedback(product_id: str, seller_id: str) -> list
  Query: db.feedback.find({ seller_id, product_id })
  Returns: list of feedback records (rating, comment, date, etc.)

9.4 get_return_trend(product_id: str, seller_id: str) -> list
  Query: returns for this product, sorted by return_date
  Returns: list of { "date": "...", "count": N } grouped by month or week
  Only meaningful if return_date field exists. Return empty list if not.

9.5 get_order_delivery_data(product_id: str, seller_id: str) -> list
  Query: db.orders.find({ seller_id, product_id })
  Returns: list of order records including delivery dates and statuses

9.6 compare_seller_products(seller_id: str) -> list
  Query: aggregate return counts across all products for this seller
  Returns: list of { "product_id": "...", "product_name": "...", "return_count": N, "return_rate": X }

9.7 detect_anomalies(product_id: str, seller_id: str) -> dict
  Logic: fetch return data over time, look for spikes, drops, or clusters
  Returns: { "anomalies_detected": True/False, "details": ["..."] }

9.8 get_sku_return_breakdown(product_id: str, seller_id: str) -> list
  Query: returns for this product grouped by sku_id or variant_attributes
  Returns: list of { "sku_id": "...", "variant": "...", "return_count": N }
  Only meaningful if sku_id is present in return/order records.
  Return empty list if sku_id field is absent from the data.

HOW TOOLS ARE REGISTERED FOR GEMINI FUNCTION CALLING:
Each tool has a corresponding JSON schema definition (name, description,
parameters) passed to the Gemini API as the "tools" parameter.
Gemini uses these definitions to decide which tools to call and with what args.

SECTION 10: AGENT PIPELINE (app/agent/pipeline.py)
--------------------------------------------------------------------------------

The pipeline is the orchestration layer between the API routes and the LLM.
It has two modes: DASHBOARD MODE and CHATBOT MODE.

10.1 DASHBOARD MODE — run_dashboard_analysis(product_id, seller_id)

  Step 1: Call a fixed set of core tools unconditionally:
    - get_product_return_data(product_id, seller_id)
    - get_return_reasons_breakdown(product_id, seller_id)
    - get_customer_feedback(product_id, seller_id)
    - get_return_trend(product_id, seller_id)
    - get_order_delivery_data(product_id, seller_id)
    - detect_anomalies(product_id, seller_id)
    - get_sku_return_breakdown(product_id, seller_id)  ← only if SKU data exists

  Step 2: Combine all tool results into a single data payload.

  Step 3: Send to Gemini with the DASHBOARD SYSTEM PROMPT (see Section 11).
    The prompt instructs Gemini to return a structured JSON with these fields:
      return_signal, summary, root_cause, patterns, evidence,
      trend, suggestions, anomalies

  Step 4: Parse Gemini's response and return the structured object.

10.2 CHATBOT MODE — run_chat(query, seller_id)

  Step 0: Scope check BEFORE calling any tool or LLM.
    Ask Gemini (or use a simple keyword match):
      "Is this query related to returns, products, orders, feedback, delivery,
       or seller analytics? Answer: IN_SCOPE or OUT_OF_SCOPE"
    If OUT_OF_SCOPE: immediately return the refusal message. No tools called.

  Step 1: Pass the query + tool definitions to Gemini with function calling enabled.
    Gemini decides which tools are relevant and returns tool_use blocks.

  Step 2: Execute only the tools Gemini requested with the args Gemini provided.
    Always inject seller_id from the JWT — never let Gemini supply the seller_id.

  Step 3: Pass raw tool results + original query back to Gemini.
    Instruct Gemini to answer ONLY from the tool results. No invented data.

  Step 4: Return Gemini's final answer and the list of tools that were called.

IMPORTANT AGENT RULES:
  - The LLM is NEVER trusted for security decisions.
  - seller_id is ALWAYS from the JWT, never from LLM output or user input.
  - Tool results are NEVER pre-summarized before sending to Gemini.
  - No caching of AI responses. Fresh data every time.
  - If a tool returns empty data, Gemini is told to note the absence,
    not to invent data or make assumptions.

SECTION 11: GEMINI PROMPTS
--------------------------------------------------------------------------------

11.1 DASHBOARD ANALYSIS PROMPT

System prompt sent with dashboard tool results:

  "You are a Retail Return Reasoning Agent.
   You are given raw data retrieved from a seller's product return records.
   Analyse the data and produce a structured JSON response with these fields:

   - return_signal: one of 'High', 'Normal', or 'Low'
   - summary: a single sentence describing the return situation for this product
   - root_cause: the primary reason returns are occurring, based on the data
   - patterns: a list of observed patterns (e.g. 'returns spike in December')
   - evidence: specific data points that support your findings
     (counts, dates, feedback excerpts, percentages)
   - trend: time-series insight if date data was provided, else omit this field
   - suggestions: at least 3 specific, actionable recommendations for the seller
   - anomalies: list of unusual patterns detected, or empty list if none

   Rules:
   - Only use the provided data. Do not use external knowledge.
   - Do not infer data that is not present. If a field is absent from the data,
     do not reference it.
   - If insufficient data exists for a field, omit that field from the response.
   - Return valid JSON only. No markdown. No explanation outside the JSON."

11.2 CHATBOT RESPONSE PROMPT

System prompt used in chatbot mode:

  "You are a Retail Return Reasoning Agent.
   You can only answer questions related to:
   returns, products, return reasons, orders, deliveries, customer feedback,
   and seller performance metrics.

   You must not answer questions outside this scope.
   If a query is outside scope, respond:
   'I can only assist with return, product, order, and feedback related questions.'

   When answering:
   - Use ONLY the data provided in the tool results.
   - Never use external knowledge.
   - Never guess or infer data that is not present.
   - If information is unavailable, say: 'Insufficient data available.'
   - Support every claim with specific data points from the tool results.
   - Be concise and specific. Sellers want actionable answers, not summaries."

SECTION 12: SECURITY RULES (non-negotiable)
--------------------------------------------------------------------------------

1. seller_id always comes from the JWT. Never from request body or query params.
2. Every DB query filters by seller_id. No exceptions.
3. Protected routes use get_current_seller as a FastAPI Depends() dependency.
4. A 401 is returned for missing, expired, or invalid tokens.
5. A seller who requests a product_id that isn't theirs gets no data back
   (the seller_id filter on the query ensures this naturally).
6. SECRET_KEY and GEMINI_API_KEY never appear in code. Always from .env.
7. .env is never committed. gitignore it.
8. The LLM cannot bypass the scope guard. Tool functions enforce filtering.

Formalized Agent Tool Specification Manifest
This is the exact structural specification required by the tools parameters configuration in the Gemini API.
1. get_product_return_data
Description: Fetches all raw, core return transaction logs for a specific parent product line scoped to the authenticated seller.
Condition / When to Call: Call when the user asks a general question about the overall return volume, frequency, or basic return entry logs of a product line.
Dependencies: None. Assumes baseline presence of product and return identifiers.
Output Scope: Real-time database fetch. All-time historical data available in the collection.
Error / Null Behavior: If no returns exist for the product, returns an empty array: [].
Return Schema:
JSON
[
  {
    "return_id": "string (ObjectId)",
    "order_id": "string (ObjectId)",
    "customer_id": "string",
    "return_date": "string (ISODate)",
    "return_reason": "string",
    "refund_amount": "number (Decimal)"
  }
]

2. get_return_reasons_breakdown
Description: Retrieves a high-level grouped categorical count breakdown of return reasons for a specified product.
Condition / When to Call: Automatically called on dashboard load or when a user explicitly asks "Why are people returning this item?" or "What is the main issue with this product?"
Dependencies: Requires the return_reason_category enum field to be populated in the dataset.
Output Scope: Real-time database aggregation. All-time historical data.
Error / Null Behavior: Returns an empty object {} if no returns match the product.
Return Schema:
JSON
{
  "product_id": "string (ObjectId)",
  "total_returns": "integer",
  "breakdown": [
    { "category": "Size Issue", "count": 24 },
    { "category": "Damaged", "count": 12 },
    { "category": "Quality Issue", "count": 4 }
  ]
}

3. get_customer_feedback
Description: Pulls written text reviews, star ratings, and submission dates left by customers for a specific product.
Condition / When to Call: Call when looking for qualitative evidence, context behind complaints, or when the user asks about customer sentiment/reviews.
Dependencies: Relies on the existence of data within the Feedback collection linked to the product.
Output Scope: Real-time database fetch. All-time historical records.
Error / Null Behavior: Returns an empty array [] if no feedback records exist.
Return Schema:
JSON
[
  {
    "feedback_id": "string (ObjectId)",
    "customer_id": "string (ObjectId)",
    "rating": "integer (Range 1-5)",
    "comment_text": "string or null",
    "submission_date": "string (ISODate)"
  }
]

4. get_return_trend
Description: Retrieves timestamped return transaction metrics grouped chronologically to identify operational shifts over time.
Condition / When to Call: Call only if the user asks about changes over time, monthly/weekly patterns, or return timelines.
Dependencies: Strict dependency on time-series records (return_date must be present in the data).
Output Scope: Real-time aggregation. Grouped by calendar month.
Error / Null Behavior: Gracefully returns an empty array [] if no date values are present or no records exist.
Return Schema:
JSON
[
  { "year_month": "2026-04", "return_count": 15 },
  { "year_month": "2026-05", "return_count": 42 }
]

5. get_order_delivery_data
Description: Fetches order transaction timelines, delivery dates, and fulfillment states for correlation analysis against returns.
Condition / When to Call: Call when investigating if shipping delays, logistics issues, or specific order batches are causing a spike in returns.
Dependencies: Requires order_date, delivery_date, and fulfilment_status fields to exist in the database collections.
Output Scope: Real-time database join/fetch over historical order records.
Error / Null Behavior: Returns an empty array [] if data fields are missing or unpopulated.
Return Schema:
JSON
[
  {
    "order_id": "string (ObjectId)",
    "order_date": "string (ISODate)",
    "delivery_date": "string (ISODate) or null",
    "fulfilment_status": "string (pending|shipped|delivered|cancelled|returned)",
    "quantity": "integer"
  }
]

6. compare_seller_products
Description: Pulls return metrics across all of the authenticated seller's listed items to support horizontal comparative analysis.
Condition / When to Call: Call when the user asks cross-product questions like "Which item has the highest return rate?" or "Compare product A with product B".
Dependencies: Cross-references the Products collection and Returns metrics mapped to the same seller_id.
Output Scope: Real-time seller inventory snapshot.
Error / Null Behavior: Returns an empty array [] if the seller has no active products.
Return Schema:
JSON
[
  {
    "product_id": "string (ObjectId)",
    "product_name": "string",
    "category_name": "string",
    "total_orders": "integer",
    "total_returns": "integer",
    "return_rate_percentage": "number"
  }
]

7. detect_anomalies
Description: Runs logical statistical scans on return variations to flag immediate spikes, drop-offs, or weird thematic clusters.
Condition / When to Call: Automatically triggered on dashboard load to build product cards or when users look for red flags or unusual activity.
Dependencies: None, though it yields richer patterns when time-series data or category fields exist.
Output Scope: Real-time computational evaluation over live dataset.
Error / Null Behavior: Returns an object indicating no anomalies found if data is flat or sparse.
Return Schema:
JSON
{
  "anomaly_detected": "boolean",
  "flags": [
    {
      "type": "string (Spike|Drop|Cluster)",
      "description": "string (e.g., 'Sudden 40% surge in Size Issue returns during May 2026')",
      "confidence_score": "number"
    }
  ]
}

8. get_sku_return_breakdown (New)
Description: Returns precise return frequencies and underlying reasons grouped explicitly by sku_id and corresponding variant options.
Condition / When to Call: Call whenever an overall product line shows high return rates and the agent needs to find out if a specific variant (e.g., Size Small vs. Size Large) is creating the issue.
Dependencies: Enforces a strict functional dependency on order and return transaction documents having an active, populated sku_id and structural attributes.
Output Scope: Real-time live database aggregation.
Error / Null Behavior: If the seller hasn't populated data down to the variant tier or there are no returns, it gracefully returns an empty array [].
Return Schema:
JSON
[
  {
    "sku_id": "string (ObjectId)",
    "sku_code": "string",
    "variant_attributes": {
      "size": "string",
      "color": "string"
    },
    "return_count": "integer",
    "primary_reason": "string"
  }
]
